# Transaction and Failure Semantics

Date: 2026-05-26
Status: Phase 0 contract

## Goals

- Preserve the Python `Prisma.tx()` context-manager user experience where feasible.
- Define batch and interactive transaction behavior through the JS bridge.
- Make rollback behavior deterministic for Python exceptions, cancellation, timeout, bridge death, and disconnect.
- Avoid claiming transaction support until provider-specific fixtures pass.

## Current Python behavior anchors

The existing runtime exposes transaction methods through the engine abstraction:

- `start_transaction(content={timeout,max_wait}) -> TransactionId`
- `commit_transaction(tx_id)`
- `rollback_transaction(tx_id)`
- Query execution carries the transaction ID on every request.

The Python context manager commits when the block exits normally and rolls back when the block exits with an exception. Existing nested transaction behavior warns that the current client is already in a transaction and may be surprising.

## Transaction modes

| Mode | Phase 0 classification | JS bridge target |
| --- | --- | --- |
| Batch transaction | Required for first parity gate | `query.batch` mapped to Prisma `$transaction([...])` where operations are independent. |
| Interactive transaction | Required before default flip unless explicitly deferred with release-note break | `transaction.start` creates a Node-side transaction client registry entry; `query.execute` includes `transactionId`. |
| Nested interactive transaction | Unsupported for first JS bridge default | Return `TRANSACTION_NESTED_UNSUPPORTED`; do not silently start nested sessions. |
| Long-lived transaction across reconnect | Unsupported | Transaction IDs are scoped to one bridge process and die with it. |

## Transaction ID contract

- `transactionId` is an opaque string generated by the Node bridge.
- IDs must not encode secrets or database connection strings.
- IDs are scoped to one bridge process.
- IDs are single-use lifecycle handles: `open -> committed` or `open -> rolled_back` or `open -> failed`.
- A closed ID must never be reused.
- Python must treat `TRANSACTION_CLOSED` as a terminal error for that transaction client copy.

## Interactive transaction lifecycle

### Start

Request:

```json
{
  "id": "req_tx_start_1",
  "method": "transaction.start",
  "params": {"timeoutMs": 5000, "maxWaitMs": 2000, "isolationLevel": null},
  "timeoutMs": 7000
}
```

Result:

```json
{"transactionId": "tx_01J...", "expiresAt": "2026-05-26T05:55:00.000Z"}
```

Rules:

- `maxWaitMs` is the maximum time to acquire the transaction.
- `timeoutMs` is the maximum transaction lifetime once acquired.
- The bridge must store the Prisma transaction client associated with `transactionId` until commit/rollback/timeout/failure.
- The bridge must reject `transaction.start` if the caller passes an existing `transactionId`.

### Query inside transaction

Every transactional query includes the transaction ID in the request envelope.

```json
{
  "id": "req_tx_query_1",
  "method": "query.execute",
  "transactionId": "tx_01J...",
  "params": {"kind": "model", "model": "User", "action": "update", "args": {"where": {"id": 1}, "data": {"balance": {"decrement": 50}}}},
  "timeoutMs": 30000
}
```

Rules:

- The bridge routes the operation through the transaction-scoped Prisma client.
- If the transaction is tainted or closed, the bridge returns `TRANSACTION_CLOSED`.
- Query timeout inside an interactive transaction taints the transaction and triggers rollback.

### Commit

Request:

```json
{"id":"req_tx_commit_1","method":"transaction.commit","transactionId":"tx_01J...","params":{},"timeoutMs":5000}
```

Result:

```json
{"status":"committed"}
```

Commit closes the transaction. Late queries with the same ID must fail.

### Rollback

Request:

```json
{"id":"req_tx_rollback_1","method":"transaction.rollback","transactionId":"tx_01J...","params":{"reason":"python-exception"},"timeoutMs":5000}
```

Result:

```json
{"status":"rolled_back"}
```

Rollback closes the transaction. Rollback is best-effort after bridge process failures; Python must surface the original exception when rollback also fails, matching current context-manager intent.

## Rollback policy

| Trigger | Required bridge behavior | Python-facing result |
| --- | --- | --- |
| Python exception inside `Prisma.tx()` block | Python calls `transaction.rollback`. | Original exception is re-raised; rollback failure is logged/attached but does not mask original by default. |
| `transaction.start` timeout before ID returned | No active transaction ID exists. | Raise timeout/startup error. |
| Query timeout inside transaction | Mark transaction tainted, attempt rollback, reject further operations. | Raise timeout/cancel error; transaction client becomes unusable. |
| Explicit cancellation inside transaction | Mark transaction tainted, attempt rollback. | Raise cancellation error; transaction client becomes unusable. |
| Bridge process exits | Python treats all open transactions as failed/unknown; process restart does not resurrect them. | Raise `BRIDGE_PROCESS_EXITED` or mapped engine connection error. |
| Python client disconnect | Python asks bridge to rollback active transactions before disconnect. | Disconnect succeeds only after rollback or after forced process termination with warning. |
| Database connection error | Bridge maps Prisma/adapter error and closes/taints transaction according to Prisma result. | Preserve Prisma code in `prismaCode`; retryable defaults false. |

## Connection pinning expectations

- Interactive transaction connection/session pinning is delegated to Prisma JS Client and the chosen driver adapter.
- The bridge must not invent its own connection pool semantics.
- Provider-specific transaction tests must prove that operations within a transaction observe uncommitted writes and rollback atomically.
- Adapter support matrix must document providers whose adapters cannot satisfy this behavior.

## Retry policy

- Automatic retries are forbidden for write operations unless a future artifact defines idempotency keys.
- Retryable startup errors may be retried by Python before a request is accepted.
- Retryable read operations are allowed only when the bridge can prove that no transaction is open and no side effects occurred.
- Prisma error codes may inform `retryable`, but the bridge must be conservative.

## Batch transaction semantics

Batch transactions are independent operation arrays.

Rules:

- The bridge must preserve operation order.
- If any operation fails, all operations roll back according to Prisma `$transaction([...])` behavior.
- Batch operations do not expose an interactive `transactionId` to Python.
- Batch support must include success, validation error, runtime error, and rollback fixtures before release.

## Nested transaction posture

Nested interactive transactions are not supported in the first JS bridge default mode.

Rationale:

- Current Python docs warn that transaction support with model-based queries is not stable.
- Existing nested behavior already warns that it can be surprising.
- Prisma JS Client nested interactive behavior and adapter savepoint semantics require provider-specific proof.

Required behavior:

- If Python attempts `Prisma.tx()` while the client copy already has `_tx_id`, JS bridge mode returns `TRANSACTION_NESTED_UNSUPPORTED` or prevents the nested start at the Python layer.
- A future savepoint-based design must be a separate reviewed contract change.

## Acceptance criteria

- Batch and interactive transaction flows are described with request/response examples.
- Rollback behavior is specified for Python exception, cancellation, timeout, bridge death, and disconnect.
- Transaction IDs and lifecycle state are defined.
- Nested transaction behavior is explicit and testable.
- Retry and timeout behavior is conservative.
- No runtime code is changed by this artifact.
